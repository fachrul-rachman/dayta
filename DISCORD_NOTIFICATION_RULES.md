Discord Notification Rules

Core Principle

Discord notification is a daily leadership alert.
It is not the source of truth.
It is not a chat feature.
It is not a manual reporting tool.

The source of truth remains the application data, especially rule-based flags and notification logs.
Discord is only a delivery channel for daily alert summaries.

The notification must be:

* concise
* readable
* grouped clearly
* focused only on meaningful findings
* suitable for Director-level reading

Do not send noisy or repetitive messages.
Do not send alerts for low-severity findings.
Do not carry old findings into a new day’s message unless that day itself generates new qualifying findings.

Notification Purpose

The purpose of the Discord notification is to give Director a clean daily alert showing which divisions and which people had meaningful reporting issues on that reporting day.

The notification should help Director quickly answer:

* which divisions need attention today
* who is involved
* what the main findings are

It should not try to explain everything in detail.
Detailed investigation belongs in the web application.

Delivery Scope

Target Audience

The Discord notification is intended for Director-level monitoring.

Channel Scope

Use one Discord channel only.
Do not split notifications by division.
Do not create multiple channels by severity.
Do not create separate Director/Admin notification channels by default.

Delivery Frequency

Send at most one Discord notification per reporting day.

Time of Delivery

The notification must be prepared and sent once per day at 19:00 application time.

Reason

This gives enough time for the day’s reporting findings to be collected before sending the summary.

Do not send earlier by default.
Do not send multiple summary messages throughout the day.

No-Notification Rule

If there are no qualifying findings for that reporting day, send nothing.

Do not send:

* “no findings” messages
* empty summaries
* health check pings
* placeholder notifications

Qualifying Findings

Only flags with severity:

* medium
* high

are eligible for Discord notification.

Low severity flags must never trigger Discord delivery.

Eligible MVP Flag Types

The current MVP eligible flag types are:

* missing_submission
* late_submission
* operational_dominance
* repetitive_input

Only include a flag if:

* it belongs to the reporting day being evaluated
* its severity is medium or high
* it was generated by the rule-based flag logic

Do not include:

* legacy/manual flags without valid rule context
* low severity flags
* findings from previous days unless regenerated for the current reporting day

Daily Scope Rule

The Discord message must only represent findings for the reporting day being evaluated.

Example:
If the system is sending the daily alert for 2026-03-23, then only findings tied to 2026-03-23 may be included.

Do not carry forward unresolved findings from 2026-03-22 into the 2026-03-23 message unless 2026-03-23 itself also produced those findings again.

Message Structure

Core Format

The Discord notification must be formatted as one clean daily summary message.

Required structure:

1. Header
2. Summary line
3. Grouped findings by division
4. People within each division
5. Comma-separated finding names per person

Message Style

The message must feel:

* neat
* concise
* executive-friendly
* readable in one scan

Emoji may be used sparingly to make the message feel less empty, but do not overuse them.

Use emojis only as light visual anchors, not decoration overload.

Recommended Header Style

Use a header like:
Daily Reporting Alert — 23 Mar 2026

This should clearly indicate the reporting day.

Summary Line

Include a short summary line below the header.

Recommended content:

* total number of divisions with findings
* total number of people with findings
* total number of findings

Example structure:
2 divisions • 5 people • 7 findings

Grouping Structure

The message body must be grouped by division first.

Under each division:

* list the affected people
* after each name, show the finding names only

Example structure:
Sales

* Budi - missing submission, repetitive input
* Sinta - late submission

Operations

* Andi - operational dominance

Do not show severity labels in the message body.
Do not show technical IDs.
Do not show timestamps per finding.
Do not show raw database language.

Ordering Rules

Order the content from most severe to less severe.

Division Ordering

Divisions must be ordered by the highest severity found inside that division.

Recommended order:

* divisions containing high findings first
* then divisions containing only medium findings

If severity is equal:

* sort by total number of qualifying findings descending
* then by division name ascending for stable ordering

Person Ordering Within Division

People inside a division must be ordered by:

* highest severity found for that person
* then total finding count descending
* then name ascending for stable ordering

Finding Ordering Within Person

When a person has multiple findings, order them by severity priority:

* high before medium
* then by business importance if needed

Recommended stable order for MVP:

* missing_submission
* late_submission
* operational_dominance
* repetitive_input

This keeps the message predictable and easy to scan.

Finding Name Display

Display finding names in a short readable format.

Recommended display mapping:

* missing_submission -> missing submission
* late_submission -> late submission
* operational_dominance -> operational dominance
* repetitive_input -> repetitive input

Use these labels exactly in the Discord message unless a future UI copy document changes them.

Formatting Rules

Allowed format example:

Daily Reporting Alert — 23 Mar 2026
2 divisions • 3 people • 4 findings

Sales

* Budi - missing submission, repetitive input
* Sinta - late submission

Operations

* Andi - operational dominance

Emoji Usage Rule

You may add light emojis in the message, for example:

* one alert emoji in the header
* one section marker emoji before division names if needed

But keep it restrained.

Good example:
🚨 Daily Reporting Alert — 23 Mar 2026
2 divisions • 3 people • 4 findings

Sales

* Budi - missing submission, repetitive input

Operations

* Andi - operational dominance

Do not add excessive emoji decoration.
Do not make the alert look playful.

Message Length Rule

Keep the message concise.
Do not turn Discord into a long report.

If there are many findings:

* still send one message
* keep grouping clean
* do not expand into long explanations
* do not include AI summary text in the Discord alert

The web application remains the place for deeper investigation.

Data Sources

The Discord notification must be built from application data, not from manual text entry.

Primary source:

* flags

Supporting sources:

* users
* divisions

Use flags to determine:

* whether a person has qualifying findings
* what the finding names are
* what the severity priority is

Use users and divisions to resolve:

* person name
* division name

Do not derive Discord findings from AI summaries.
Do not derive Discord findings from raw guesswork.
Do not include findings that are not represented in the rule-based flag data.

Notification Generation Workflow

Daily Job

A scheduled job must run once per day at 19:00 application time.

Its responsibilities:

1. Determine the reporting day being evaluated
2. Query all qualifying medium/high flags for that reporting day
3. Group the results by division and then by person
4. Build the final Discord message
5. Send the message to the configured Discord channel if there is at least one qualifying finding
6. Record the result in notification logs
7. Retry automatically on failure up to the allowed retry limit

No Data Case

If no qualifying findings are found:

* do not send any Discord message
* optionally log a no-send result only if needed for internal traceability
* do not clutter the channel with empty messages

Deduplication Rule

Only one Discord daily alert may be sent for the same reporting day.

The system must prevent duplicate sends for the same day in normal operation.

If the scheduled job is triggered again for the same reporting day:

* do not create a second sent notification for the same content/day unless explicitly running a controlled resend workflow in the future
* use log checks and idempotent safeguards

Retry Rule

If sending fails, the system must retry automatically up to 2 times.

Retry policy:

* original attempt
* retry attempt 1
* retry attempt 2

After that:

* mark the notification as failed
* preserve the failure details in notification history

Do not retry indefinitely.

Notification Logging

Core Principle

Every send attempt must be logged in detail.

Admin must be able to view notification history in the UI.

The notification log must support:

* operational traceability
* error review
* resend investigation if needed later

Required Log Contents

Each notification log record should capture at minimum:

* reporting day
* channel target
* delivery status
* full message content
* total number of divisions included
* total number of people included
* total number of findings included
* attempt count
* sent_at if successful
* failed_at if failed
* error_message if failed
* created_at
* updated_at

Recommended Status Values

Use clear delivery statuses such as:

* pending
* sent
* failed

If retry attempts are tracked separately, keep them visible through attempt count and timestamps.

Admin Notification History UI

Admin must be able to view detailed notification history in the application UI.

The UI should show at minimum:

* reporting day
* status
* sent time
* attempt count
* total findings
* message content
* error message if failed

The admin history view is read-only by default.
It is for monitoring the delivery process, not for manual message editing.

Admin History Purpose

Admin should be able to answer:

* was today’s alert sent
* what exactly was sent
* how many findings were included
* did delivery fail
* how many attempts were made
* what was the error

Discord Configuration

Configuration must be environment-based.
Do not hardcode webhook URLs, tokens, or channel details.

Required environment direction:

* Discord notification enabled flag
* Discord webhook URL or provider configuration
* optional application timezone if needed for scheduling consistency

Suggested Environment Variables

Recommended variables:

* DISCORD_NOTIFICATIONS_ENABLED
* DISCORD_WEBHOOK_URL

Optional if needed:

* DISCORD_NOTIFICATION_CHANNEL_NAME
* DISCORD_NOTIFICATION_TIMEOUT

Do not hardcode secrets in code or seeders.

Failure Handling Rules

If Discord delivery fails:

* retry automatically up to 2 times
* log each attempt or preserve attempt history clearly
* keep the final status accurate
* do not silently swallow the error

If Discord is disabled by configuration:

* do not send
* do not fake a successful send
* show a clear internal status or skip path in the job logic

Do Not Include in Discord

Do not include:

* low severity findings
* previous day findings that do not belong to the current reporting day
* raw database IDs
* AI narrative summaries
* full explanations or long descriptions
* admin-only data
* user emails
* timestamps per person
* internal technical terms

Do Not Do These

* do not send more than one daily summary per day
* do not send multiple messages split by division
* do not send one message per person
* do not notify for low severity
* do not carry yesterday’s findings into today’s message unless regenerated today
* do not expose severity labels in the body
* do not let users manually trigger the Discord message by default
* do not make Discord the source of truth
* do not omit logging
* do not omit retry handling

Final Notification Intent

The Discord alert must behave like a daily executive warning summary:

* one message
* one channel
* only meaningful findings
* grouped by division
* readable in seconds
* backed by full logs in the application

It must help Director notice today’s issues quickly, while the application remains the place for full investigation and follow-up.
